# Ralph Prompt Builder - 互動式準備流程

## 使用方法

在執行 `/ralph-loop` 之前，先通過這個框架準備你的 prompt。

---

## 第一步：任務定義 (Task Definition)

### 1.1 核心目標 (Core Goal)
**用一句話描述你要完成什麼：**
```
範例：實現遠程協助功能，讓用戶可以看到誰在控制他們的 session
```

### 1.2 成功標準 (Success Criteria)
**如何判斷任務完成？（必須可測試）**
```
範例：
- [ ] 用戶登錄後可以看到遠程連接列表
- [ ] 列表顯示控制者的郵箱/名稱
- [ ] 主客戶端可以接受/拒絕控制請求
- [ ] 所有功能通過 E2E 測試
```

### 1.3 完成承諾 (Completion Promise)
**選擇一個簡短的完成標記：**
```
範例：REMOTE_ASSIST_COMPLETE
```

---

## 第二步：範圍控制 (Scope Control)

### 2.1 包含什麼 (In Scope)
```
範例：
✅ 登錄接口改進
✅ Session 標識系統
✅ 控制請求處理
✅ TUI 顯示優化
```

### 2.2 不包含什麼 (Out of Scope)
```
範例：
❌ 權限管理系統
❌ 加密通訊
❌ 多人協作（>2人）
❌ 歷史記錄功能
```

### 2.3 優先級 (Priority)
```
P0 (必須): Session 標識、基本顯示
P1 (重要): 控制請求處理
P2 (可選): UI 優化、錯誤提示
```

---

## 第三步：驗證策略 (Validation Strategy)

### 3.1 測試方法
```
範例：
- 單元測試：Session 管理邏輯
- 集成測試：登錄 → 連接 → 控制流程
- E2E 測試：完整用戶場景
```

### 3.2 驗證檢查點
```
每次迭代後檢查：
1. 代碼編譯通過
2. 測試全部通過
3. TUI 可以正常啟動
4. 日誌無錯誤
```

---

## 第四步：迭代策略 (Iteration Strategy)

### 4.1 階段劃分
```
範例：
階段 1 (迭代 1-3): 實現 Session 標識系統
階段 2 (迭代 4-6): 實現控制請求處理
階段 3 (迭代 7-10): UI 優化和測試
```

### 4.2 失敗處理
```
如果測試失敗：
1. 檢查日誌找出錯誤
2. 修復問題
3. 重新運行測試
4. 記錄問題到 ISSUES.md
```

---

## 第五步：生成 Ralph Prompt

基於以上信息，生成最終的 Ralph prompt：

```markdown
# 任務：[核心目標]

## 目標
[詳細描述]

## 成功標準
- [ ] 標準 1
- [ ] 標準 2
- [ ] 標準 3

## 實現步驟
1. [步驟 1]
2. [步驟 2]
3. [步驟 3]

## 驗證
每次迭代後：
1. 運行測試：`[測試命令]`
2. 檢查日誌：`[日誌位置]`
3. 驗證功能：[驗證方法]

## 完成條件
當所有成功標準達成時，輸出：
<promise>[完成標記]</promise>

## 約束
- 範圍：[In Scope 列表]
- 不做：[Out of Scope 列表]
- 優先級：[P0/P1/P2]
```

## 範例：改進你的第一段 Prompt

### 原始版本（問題）
```
咁我哋我哋呢度呢首先我想做好嗰個远程嘅呢個协助嘅呢方面嘅嘢...
（太多細節，缺乏結構）
```

### 改進版本
```markdown
# 任務：實現遠程協助 Session 管理

## 目標
為 TUI 應用添加遠程協助功能，讓用戶可以：
1. 看到誰正在遠程控制他們的 session
2. 接受或拒絕控制請求
3. 在列表中顯示控制者信息

## 成功標準
- [ ] 登錄後可以獲取遠程連接信息
- [ ] Session 有唯一標識符
- [ ] TUI 顯示控制者郵箱/名稱
- [ ] 主客戶端可以處理控制請求
- [ ] 所有功能通過集成測試

## 實現步驟
1. 添加 Session ID 生成和管理
2. 實現遠程連接信息接口
3. 更新 TUI 顯示邏輯
4. 添加控制請求處理
5. 編寫測試驗證功能

## 驗證
每次迭代後：
1. 運行測試：`cargo test --features remote`
2. 啟動 TUI：`cargo run`
3. 檢查日誌：`tail -f logs/app.log`
4. 驗證：登錄 → 查看列表 → 測試控制請求

## 完成條件
當所有測試通過且 TUI 正常顯示遠程信息時，輸出：
<promise>REMOTE_SESSION_COMPLETE</promise>

## 約束
- 範圍：Session 管理、基本顯示、控制請求
- 不做：權限系統、加密、多人協作
- 優先級：P0=Session標識+顯示, P1=控制請求, P2=UI優化
```

## 使用建議

### 何時使用 Ralph Loop？
✅ **適合：**
- 明確的功能開發
- 需要多次迭代改進
- 有清晰的測試標準
- 綠地項目（新功能）

❌ **不適合：**
- 需要人工判斷的設計決策
- 一次性操作
- 不明確的成功標準
- 生產環境調試

### 迭代次數建議
- 簡單功能：`--max-iterations 5`
- 中等複雜度：`--max-iterations 10`
- 複雜功能：`--max-iterations 20`
- 大型重構：`--max-iterations 30`

### Completion Promise 建議
- 使用全大寫
- 簡短明確
- 反映任務本質
- 範例：`TESTS_PASS`, `FEATURE_COMPLETE`, `REFACTOR_DONE`
