# Ralph Prompt 範例：改進你的原始 Prompts

## 範例 1：遠程協助功能（改進版）

### 原始版本的問題
```
❌ 太多細節混在一起
❌ 缺乏優先級
❌ 沒有明確的完成標準
❌ 缺少驗證步驟
```

### 改進版本

```markdown
# 任務：實現遠程協助 Session 管理和顯示

## 目標
為 TUI 應用添加遠程協助功能，讓用戶可以：
1. 看到誰正在遠程控制他們的 session
2. 接受或拒絕控制請求
3. 在列表中清晰顯示控制者信息（郵箱/名稱）

## 成功標準
- [ ] 用戶登錄後可以獲取遠程連接信息
- [ ] 每個 Session 有唯一標識符（UUID）
- [ ] TUI 列表顯示控制者的郵箱或名稱
- [ ] 主客戶端可以接受/拒絕控制請求
- [ ] 控制請求有超時機制（30秒）
- [ ] 所有功能通過集成測試

## 範圍
### 包含 (In Scope)
✅ Session ID 生成和管理
✅ 遠程連接信息接口
✅ TUI 顯示邏輯更新
✅ 控制請求處理（接受/拒絕）
✅ 基本錯誤處理

### 不包含 (Out of Scope)
❌ 權限管理系統
❌ 端到端加密
❌ 多人協作（>2人同時控制）
❌ 歷史記錄功能
❌ 聊天功能

## 優先級
### P0 (必須完成)
- Session 標識系統
- 基本連接信息顯示
- 控制請求基本處理

### P1 (重要)
- 用戶信息顯示（郵箱/名稱）
- 請求超時機制
- 錯誤提示

### P2 (可選)
- UI 美化
- 動畫效果
- 詳細日誌

## 實現步驟
1. **Session 管理**
   - 添加 Session ID 生成（UUID v4）
   - 實現 Session 存儲和查詢
   - 添加 Session 生命週期管理

2. **遠程連接接口**
   - 設計 API 端點：`GET /api/sessions/:id/remote`
   - 返回控制者信息（user_id, email, name）
   - 添加認證檢查

3. **TUI 顯示更新**
   - 修改列表組件顯示遠程信息
   - 添加控制者標識（圖標 + 名稱）
   - 實現實時更新（WebSocket 或輪詢）

4. **控制請求處理**
   - 實現請求發送邏輯
   - 添加接受/拒絕處理
   - 實現超時機制（30秒）

5. **測試和驗證**
   - 編寫單元測試
   - 編寫集成測試
   - 手動測試完整流程

## 驗證策略
### 測試方法
- 單元測試：Session 管理邏輯、API 端點
- 集成測試：登錄 → 連接 → 控制請求流程
- E2E 測試：完整用戶場景（兩個客戶端）

### 每次迭代後驗證
1. 運行測試：`cargo test --features remote`
2. 檢查編譯：`cargo check`
3. 啟動 TUI：`cargo run`
4. 檢查日誌：`tail -f logs/app.log`
5. 手動驗證：
   - 登錄兩個客戶端
   - 發起遠程控制請求
   - 檢查列表顯示
   - 測試接受/拒絕功能

### 失敗處理
如果測試失敗：
1. 查看錯誤日誌：`logs/error.log`
2. 檢查 Session 狀態：`logs/session.log`
3. 修復問題
4. 重新運行測試
5. 記錄問題到 `ISSUES.md`

## 迭代階段
- **階段 1 (迭代 1-3)**: 實現 Session 標識系統和基本存儲
- **階段 2 (迭代 4-6)**: 實現遠程連接接口和 TUI 顯示
- **階段 3 (迭代 7-9)**: 實現控制請求處理和超時機制
- **階段 4 (迭代 10)**: 測試、修復 bug、優化

## 完成條件
當所有成功標準達成且測試全部通過時，輸出：
<promise>REMOTE_SESSION_COMPLETE</promise>

## 執行指令
```bash
/ralph-loop "$(cat remote_assist_prompt.md)" --max-iterations 10 --completion-promise "REMOTE_SESSION_COMPLETE"
```
```

## 範例 2：Bots 系統優化（改進版）

### 原始版本的問題
```
❌ 範圍太大（5個 bots + docker + skills + mcps）
❌ 目標模糊（"更好"、"更完整"）
❌ 沒有具體的驗證步驟

```

### 改進版本：拆分成多個 Ralph Loop

#### Loop 1: Bots 任務執行穩定性

```markdown
# 任務：提升 Bots 任務執行穩定性

## 目標
優化 5 個 bots 的任務執行流程，確保：
1. 任務獲取不中斷
2. 任務執行不錯亂
3. 任務完成有記錄
4. 失敗有重試機制

## 成功標準
- [ ] Bots 可以穩定從隊列獲取任務
- [ ] 任務執行過程有完整日誌
- [ ] 任務完成後正確提交狀態到數據庫
- [ ] 失敗任務自動重試（最多3次）
- [ ] 系統運行 24 小時無崩潰
- [ ] 任務成功率 > 85%

## 範圍
### 包含 (In Scope)
✅ 任務隊列穩定性
✅ 執行流程優化
✅ 日誌系統完善
✅ 錯誤處理和重試
✅ 狀態提交機制

### 不包含 (Out of Scope)
❌ 新功能開發
❌ UI 改進
❌ 性能優化（除非影響穩定性）
❌ Docker 配置調整
❌ Skills/MCPs 更新

## 優先級
### P0 (必須完成)
- 任務獲取穩定性
- 基本錯誤處理
- 狀態提交機制

### P1 (重要)
- 重試機制
- 完整日誌
- 崩潰恢復

### P2 (可選)
- 性能監控
- 詳細統計

## 實現步驟
1. **分析現有問題**
   - 檢查日誌找出常見錯誤
   - 統計失敗率和失敗原因
   - 識別不穩定的環節

2. **優化任務獲取**
   - 添加連接池管理
   - 實現心跳機制
   - 添加超時處理

3. **完善錯誤處理**
   - 捕獲所有異常
   - 分類錯誤類型（可重試 vs 不可重試）
   - 實現重試邏輯（指數退避）

4. **改進日誌系統**
   - 統一日誌格式
   - 添加任務 ID 追蹤
   - 記錄關鍵執行步驟

5. **優化狀態提交**
   - 確保事務性
   - 添加失敗重試
   - 實現冪等性

## 驗證策略
### 測試方法
- 單元測試：錯誤處理、重試邏輯
- 集成測試：完整任務執行流程
- 壓力測試：並發 100 個任務
- 穩定性測試：運行 24 小時

### 每次迭代後驗證
1. 運行測試：`pytest tests/bots/`
2. 檢查日誌：`docker logs bot-1 | grep ERROR`
3. 查看統計：`python scripts/analyze_logs.py`
4. 手動測試：
   - 提交 10 個測試任務
   - 觀察執行過程
   - 檢查成功率
   - 驗證日誌完整性

### 成功指標
- 任務成功率：從當前 ~60% 提升到 >85%
- 系統穩定性：24 小時無崩潰
- 日誌完整性：100% 任務有完整日誌
- 重試成功率：失敗任務 70% 通過重試成功

## 數據驅動改進
每次迭代分析：
1. 查看數據庫統計：
   ```sql
   SELECT status, COUNT(*) FROM tasks
   WHERE created_at > NOW() - INTERVAL '1 hour'
   GROUP BY status;
   ```

2. 分析失敗原因：
   ```bash
   python scripts/analyze_failures.py --last-24h
   ```

3. 識別高失敗率網站：
   ```bash
   python scripts/failure_by_domain.py
   ```

4. 根據數據調整策略

## 迭代階段
- **階段 1 (迭代 1-2)**: 分析問題，識別根本原因
- **階段 2 (迭代 3-5)**: 實現錯誤處理和重試機制
- **階段 3 (迭代 6-8)**: 優化日誌和狀態提交
- **階段 4 (迭代 9-10)**: 壓力測試和穩定性驗證

## 完成條件
當任務成功率 >85% 且系統運行 24 小時無崩潰時，輸出：
<promise>BOTS_STABILITY_COMPLETE</promise>

## 執行指令
```bash

```
```

## 使用建議

### 執行順序
1. 先用 `ralph_prompt_builder.py` 準備 prompt
2. 檢查生成的 prompt 是否符合要求
3. 執行 Ralph Loop
4. 觀察迭代過程
5. 根據結果調整 prompt（如果需要）

### 調試技巧
如果 Ralph Loop 沒有按預期工作：
1. 檢查 `<promise>` 標記是否正確
2. 確認成功標準是否可測試
3. 查看日誌找出卡住的地方
4. 考慮降低範圍或增加迭代次數

### 最佳實踐
- **小步快跑**: 寧可多個小 Loop，不要一個大 Loop
- **數據驅動**: 用日誌和統計指導改進
- **明確標準**: 成功標準必須可測試、可量化
- **階段清晰**: 每個階段有明確的輸入和輸出
